# ADR-0026 — Search Pipeline Improvements: Chunking + Query Expansion

**Status:** Accepted
**Date:** 2026-03
**Supersedes:** Extends ADR-0005 (Hybrid Search with RRF)

## Context

Research into qmd (github.com/tobi/qmd) — a local-first markdown search engine — identified
techniques not yet adopted in innie's search stack. qmd uses the same foundational architecture
(SQLite FTS5 + sqlite-vec + RRF) but adds:

1. Markdown-aware chunking (~900 tokens, header-boundary-aware)
2. Query expansion (LLM generates 1-2 alternate phrasings before searching)
3. LLM reranking (cross-encoder scores top results 0-10)

Analysis: innie's 100-word sliding window creates overly granular chunks that fracture
semantically coherent document sections. The single-query approach has recall gaps when
vocabulary between indexed content and query terms diverges.

## Decision

Adopt (1) and (2). Defer (3).

### Chunking

New defaults: 300 words per chunk, 60-word overlap (20%). Add markdown-aware splitting:
split on `## / ###` headers first; apply word-window fallback only for sections > chunk_words.
When a section is split, prefix each sub-chunk with the header text for context.

Config changes:
- `index.chunk_words`: 100 → 300
- `index.chunk_overlap`: 15 → 60
- `index.chunk_markdown_aware`: (new) true

### Query Expansion

New `_expand_query(query)` function calls an LLM to generate one alternative phrasing.
Uses `search.expansion_model` config (defaults to `heartbeat.model`). 10s timeout.
Silently skips on any failure — graceful degradation maintained.

Modified `search_hybrid()`: runs RRF with original query at 2x weight + alt query at 1x weight.

Config changes:
- `search.query_expansion`: (new) false (opt-in)
- `search.expansion_model`: (new) "auto" (uses heartbeat model)

### Reranking — Deferred

Qwen3-reranker 0.6B adds precision by re-scoring top results with a cross-encoder. Deferred
because: (a) requires a new model in vLLM or Docker, (b) chunking + expansion improvements
are likely sufficient first, (c) infra decision best made after observing quality gains from
phases 1 and 2. Track as future ADR-0027.

## Rationale

**Chunk size:** 100 words (~75 tokens) is below the threshold where embeddings capture enough
context to be useful. BGE-base-en-v1.5 was designed for inputs up to 512 tokens. 300 words
(~225 tokens) hits the useful range while keeping chunks small enough for precision.
Markdown-aware splitting preserves topic coherence that word-window sliding destroys.

**Query expansion:** Vocabulary mismatch is a known failure mode for hybrid search. If you
documented "authentication" and search for "login", FTS5 misses it; semantic may catch it,
but only if the embedding space aligns. One alternate phrasing generated by an LLM dramatically
improves recall at low cost. Using the existing heartbeat model means no new service required.

**Opt-in default for expansion:** Adds 1-2s latency per search. Acceptable at session start;
potentially annoying for interactive `innie search`. Default off lets users enable when ready.

## Consequences

**Positive:**
- Larger chunks → richer context per result, better reranking surface for future phase 3
- Markdown-aware split → chunks align with document structure/topics
- Query expansion → meaningful recall improvement for vocabulary mismatch
- All improvements are additive; graceful degradation preserved throughout
- Existing index rebuilt on next `innie index` — no migration needed, state/ is ephemeral

**Negative:**
- Index must be rebuilt after chunking changes (old 100-word chunks must be discarded)
- Query expansion requires LLM endpoint; silent no-op if unconfigured
- Slightly larger DB (fewer chunks, but larger per-chunk content)

**Neutral:**
- `index.chunk_markdown_aware = false` restores previous behavior (escape hatch)
- `search.query_expansion = false` (default) means zero behavior change without opt-in
